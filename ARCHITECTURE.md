# Архітектура проекту Texnoplus Service

Цей документ описує загальну архітектуру, технологічний стек та взаємодію компонентів у проекті сервісного центру Техноплюс.

## Загальна структура

Проект складається з двох основних частин, які працюють разом:
1. **Публічний веб-сайт (Клієнтська частина)** - сайт для клієнтів з інформацією про послуги та можливістю перевірити статус ремонту.
2. **Внутрішня CRM система (Desktop)** - програма для менеджерів та майстрів сервісного центру для обліку ремонтів.

Між ними налаштована синхронізація через хмарну базу даних Firebase.

---

## 1. Публічний веб-сайт (Frontend)

Веб-сайт побудований за принципом Single Page Application (SPA) з використанням сучасного стеку React.

### Технологічний стек
* **Фреймворк:** React 18 + TypeScript
* **Збирач (Bundler):** Vite
* **Маршрутизація:** React Router v6
* **Стилізація:** Tailwind CSS
* **UI Компоненти:** Radix UI (shadcn/ui підхід), Framer Motion для анімацій, Lucide React для іконок
* **SEO та Pre-rendering:** `react-snap` (генерує статичний HTML під час збірки для кращої індексації пошуковими системами)

### Структура маршрутизаторів (Routes) - `src/App.tsx`
* `/` - Головна сторінка (`HomePage`)
* `/faq` - Часті питання (`FaqPage`)
* `/contact` - Контакти (`ContactPage`)
* `/brands` - Бренди (`BrandsPage`)
* `/service/:id` - Сторінка конкретної категорії послуг (напр. ремонт телефонів) (`ServicePage`)
* `/service/:id/:brandSlug` - Сторінка послуг для конкретного бренду
* `/status` - Перевірка статусу ремонту (`StatusPage`)
* `/*` - Сторінка 404 (`NotFoundPage`)

### Джерела даних
* Каталог послуг, ціни та бренди завантажуються динамічно з файлу `public/data/services.json` при ініціалізації додатку.
* Статус ремонту (на сторінці `/status`) підтягується безпосередньо з **Firebase Realtime Database**.

---

## 2. Внутрішня CRM система (Backend / Desktop)

Локальна програма, яка використовується безпосередньо в сервісному центрі для ведення обліку.

### Технологічний стек
* **Мова:** Python 3
* **GUI Фреймворк:** PyQt5 (`app2.py`)
* **База даних:** SQLite (`service_center.db`)
* **Машинне навчання (ML):** scikit-learn (`price_model.pkl` для прогнозування цін на ремонти)

### Основні функції (`app2.py`)
* Реєстрація нових клієнтів та пристроїв.
* Ведення статусу ремонту (Прийнято, В ремонті, Готово, Видано тощо).
* Формування квитанцій для друку.

---

## 3. Синхронізація та Інтеграція

Щоб клієнти могли бачити актуальний статус свого ремонту на сайті, локальна база даних CRM синхронізується з хмарою.

1. **База даних:** Усі ремонти зберігаються локально в `service_center.db`.
2. **Синхронізація:** Скрипт `sync_all.py` бере дані з локальної SQLite та завантажує їх у **Firebase Realtime Database**. Для цього використовується сервісний акаунт Firebase (файл `texnoplus-service-firebase-adminsdk-*.json`).
3. **Відображення клієнту:** Користувач на сайті (роут `/status`) вводить номер квитанції та останні 4 цифри телефону. Сайт робить запит у Firebase та показує поточний статус.

---

## 4. Деплой (Розгортання)

* **Збірка:** Використовується `npm run build`, який компілює TypeScript (`tsc`) та збирає оптимізовані файли через Vite.
* **Pre-rendering:** Після збірки запускається `react-snap` (`postbuild` скрипт), який проходиться по всіх роутах та зберігає їх як статичні HTML-файли у папку `dist/`.
* **Docker:** Проект містить `Dockerfile`. Збірка відбувається у два етапи (multi-stage build):
  1. `builder` (Node.js) - встановлює залежності, включаючи Chromium для `react-snap`, та збирає проект.
  2. `production` (Nginx) - бере готові файли з `dist/` і обслуговує їх через легкоатлетний веб-сервер Nginx (`nginx.conf`).

## Додаткові скрипти
* `generate-sitemap.cjs` - автоматично генерує `sitemap.xml` для SEO перед збіркою.
* Скрипти для скрапінгу та автоматизації (`scrape-wvl.cjs`, `fetch-logos.cjs` тощо).
